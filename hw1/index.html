Task 1: Drawing Single-Color Triangles

Generally, rasterization takes as input 3 points of a triangle, and the output is a set of pixel values approximating the triangle. To start, I found the min and max x and y values of the triangle. This way, when we are iterating through pixels, we will not be looking at points that are out of bounds (so we'll save a lot of time!). Then, I iterated through all the x and y values between the respective minimums and maximums, so all the possible pixel values. For each pair of (x,y), I checked if it was inside the triangle with my helper function. My helper function, inside, is an implementation of the three line test, where I take in the triangle values as well as the point we are checking. We are checking if (x,y) is in triangle t. I implemented this by iterating through all the pairs of triangle points, and using the formulas from lecture, I found dx and dy, and plugged those into the Line equation, Li. I kept track of the Lis for all the pairs of points, and then at the end, I checked if they were all greater than or equal to 0, and if so, return true (so the point is inside the triangle). However, when I did this, basic test 6 failed, so I also added a check to see if they were all less than or equal to 0, and this would also return true. This way, I accounted for winding order of the vertices, aka the direction the points were going in. Finally, back in the rasterize_triangle function, if the current point (x,y) was inside the triangle, I used fill_pixel to fill that pixel with the color (which was a parameter). Since I used minimum and maximum values in my iterations, I ensured that I had a reasonable bounding box aka I only search through the smallest rectangle that can be drawn while ensuring the entire triangle is in it.

Here are the images generated with the rasterizer.

Extra credit: To make my rasterizer run faster, I implemented incremental triangle traversal. If you notice with the line equation from lecture, if we make a change to the x or y in L(x,y), the value changes by a constant. So instead of recomputing the line values every iteration of the for loop, we can do it once, and then just add the constant. Also, I iterated via y first and then x, since x is the number of columns and y is the number of rows. Via the data, we can see that implementing this did speed up the rasterizer a bit.
